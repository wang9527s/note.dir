## Git commit id / Gerrit Change-Id 知识点

### 背景问题

1. 发现不同机器上，同一个提交看起来对应的 commit id 不同；思考commit id、change id的问题。

### 原因 / 结论

1. commit id 是由 commit 对象内容计算出来的哈希值。
   - commit 对象通常包含：
     - 代码快照对应的 `tree`
     - 父提交 `parent`
     - 作者 `author`
     - 提交者 `committer`
     - author / committer 时间戳
     - commit message
   - 同一个 commit 对象，在不同机器、不同 clone 仓库中的 commit id 一定相同。
      - 刚刚clone的时候，commit id，实际上是之前push的时候，推送端的commit id。所以刚刚clone的时候 commit id一致
      - 除非本地有rebase等操作，才会导致两个文件夹中的git clone，commit 不同

#### Change-Id

1. Gerrit 的 Change-Id。
   - Change-Id 是 Gerrit 用来关联同一个 review 的标识。
   - 一个 Change-Id 可以对应多个不同的 commit id，也就是多个 patch set。
   - 所以重新 push 后，commit id 可能变化，但 Change-Id 不变，Gerrit 仍然认为这是同一个 change 的新 patch set。

2. `git cherry-pick -m 1 <merge_commit_id>` 的含义：

     假设有这个提交图：

     ```text
     A --- B --- C        main
      \         /
       D --- E            feature
     ```

     这里 `C` 是一个 merge commit，它有两个 parent：

     ```text
     parent 1 = B
     parent 2 = E
     ```

     也就是说：

     ```text
     C = 把 E 分支 merge 到 B 上之后的结果
     ```

     普通 commit 只有一个 parent，所以 Git 知道这个 commit 引入了什么变化：

     ```text
     变化 = 当前 commit - parent
     ```

     但 merge commit 有两个 parent，Git 不知道你想按哪个 parent 来算变化：

     ```text
     C - B ?   # 相对于 main 主线，merge 引入了什么
     C - E ?   # 相对于 feature 分支，merge 引入了什么
     ```

     所以需要 `-m 1` 告诉 Git：

     ```bash
     git cherry-pick -m 1 C
     ```

     意思是：

     ```text
     以 C 的第 1 个 parent，也就是 B，作为主线。
     计算 C 相对于 B 的变化。
     然后把这部分变化应用到当前分支。
     ```

   - 本地常见 merge 习惯是 `git checkout 主线 && git merge 特性分支`，这种情况下 merge commit 的第 1 个 parent 通常就是主线原来的 HEAD，所以 cherry-pick 这个 merge commit 时一般用 `-m 1`。
